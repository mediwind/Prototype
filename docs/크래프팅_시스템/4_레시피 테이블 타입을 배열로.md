# 레시피 테이블 타입을 배열로

---

## Recipe의 table_types를 배열(Array) 방식으로 변경한 이유

### 1. 여러 종류의 테이블에서 동일한 레시피 사용 가능

- 하나의 레시피가 여러 제작 테이블(예: 모루, 용광로 등)에서 동시에 사용될 수 있습니다.
- 예를 들어, "철 주괴" 레시피를 모루와 용광로 모두에서 제작 가능하게 하려면
table_types에 `["anvil", "furnace"]`처럼 여러 값을 넣으면 됩니다.

### **2. 레시피 데이터의 중복 방지**

**(배열이 아닐 때의 문제점)**

- 예를 들어 "철 주괴 만들기" 레시피가 **모루(anvil)**와 **용광로(furnace)** 두 곳에서 모두 제작 가능해야 한다고 가정해봅시다.
- 만약 `@export var table_type: String`(단일 타입)만 있다면:
    - **anvil용 레시피**:
        - 이름: "Iron Ingot"
        - table_type: "anvil"
        - 재료: 철광석 3개
    - **furnace용 레시피**:
        - 이름: "Iron Ingot"
        - table_type: "furnace"
        - 재료: 철광석 3개
- **결과:**
    - 내용이 완전히 같은 레시피 리소스 파일을 두 개 만들어야 함(데이터 중복).

**(배열로 확장했을 때)**

- `@export var table_types: Array[String] = ["anvil", "furnace"]`
- **Iron Ingot 레시피 하나만 만들고,**
    - table_types에 "anvil", "furnace"를 모두 넣으면
    - 두 테이블에서 모두 이 레시피가 뜸.
- **결과:**
    - 레시피 리소스 파일이 하나만 필요(중복 없음).

### **3. 유지보수 및 확장성 향상**

**(배열이 아닐 때의 문제점)**

- 나중에 "야외 캠프파이어(campfire)"라는 새로운 테이블이 추가되어"철 주괴"도 여기서 만들 수 있게 하고 싶음.
- 단일 타입 구조라면:
    - campfire용 레시피를 새로 복사해서 만들어야 함.
    - 레시피 내용이 바뀌면(예: 재료 변경)→ anvil, furnace, campfire용 레시피를 모두 수정해야 함(수정 누락 위험).

**(배열로 확장했을 때)**

- 기존 Iron Ingot 레시피의 table_types에`"campfire"`만 추가하면 됨.
- 레시피 내용(재료, 결과물 등)을 바꿔도**하나의 리소스만 수정하면 모든 테이블에 반영됨.**
- **확장/수정이 훨씬 쉽고, 실수도 줄어듦.**

### 4. 게임 밸런싱 및 콘텐츠 확장에 유리

- 특정 레시피를 여러 제작 방식(예: 야외 캠프파이어, 주방, 용광로 등)에서
동시에 제공하고 싶을 때, 배열 방식이 훨씬 유연하게 대응할 수 있습니다.

---

**요약:**

배열 방식(table_types: Array[String])은

레시피의 재사용성, 데이터 중복 방지, 유지보수/확장성,

그리고 게임 콘텐츠 확장에 모두 유리한 구조입니다.

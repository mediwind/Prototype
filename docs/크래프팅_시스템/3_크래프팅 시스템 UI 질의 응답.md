# 크래프팅 시스템 : UI 질의 응답

# 질문

크래프팅 시스템을 계속 이어서 완성해보자.

1. #file:recipe.gd #file:recipe_ingredient.gd 크래프팅 시스템을 위한 리소스(데이터)를 정의했어.
2. #file:crafting_ui.tscn 크래프팅 테이블과 상화작용할 때 팝업될 크래프팅 UI 씬을 생성했어.

이제 크래프팅 테이블 씬을 만들려 해.

1. 크래프팅 테이블에 플레이어가 가까이 접근해 특정 키나 마우스 버튼으로 클릭시 크래프팅UI가 팝업되기를 원해.
2. 크래프팅 테이블은 여러 종류로 나눌 예정이야. 종류와 텍스처는 바뀔 수 있지만 일단은 무기를 만들 수 있는 모루, 갑옷을 만들 수 있는 선반, 광물을 녹여 주괴를 만드는 용광로 등으로 나눌 예정이야.
3. 크래프팅 테이블의 종류에 따라 RecipeList(ItemList 노드)에서 뜨는 레시피 후보들이 각기 달랐으면 좋겠어.
4. 3번과 같이 크래프팅 테이블의 종류에 따라 접근할 수 있는 레시피들을 구분할 때 #file:item_data.gd #file:equipment_data.gd #file:consumable_data.gd 혹은 Recipe나 RecipeIngredient를 기준으로 삼아야 할지, 아니면 쉽게 구분을 구현할 수 있는 방법이 있는지 알고 싶어.
5. 나중에 개발이 진행되며 테이블 종류를 추가하거나 기존의 테이블의 담당 레시피들을 변경하는 경우 손쉽게 유지보수할 수 있으면 좋겠어.
6. #file:slot.gd #file:slot.tscn 크래프팅 테이블에서 재료 아이템과 결과물 아이템을 현재는 Slot씬에 담아 표현하는 중인데 Slot씬의 스크립트 기능들은 크래프팅 테이블에서는 필요가 없을 것 같아서 Slot 씬을 instantiate하되 붙어 있는 스크립트는 제거(없이)해서 사용하면 어떨까 하는데 어때
7. 크래프팅 UI를 팝업했을 때 RecipeList에서 제작 가능한 아이템들(플레이어 인벤토리에 재료가 있고 제작 결과물을 담을 슬롯이 있는 경우)이 아닌 경우는 이름 텍스트가 GreyedOut되기를 원해.

일단 현재 구현하고 싶은 내용은 위와 같아. 가능성은 물론 유지보수/확장 측면에서도 평가해줘. 구현하는 경우 스크립트를 CraftingUI의 루트 노드인 CraftingUI(PanelContainer)에 붙여 모든 것을 담당하도록 할지 하위 자식 노드들마다 역할을 분담시킬지 아니면 다른 방법이 있을지를 고민하고 제안해줘.

답변시 현재는 구체적인 코드보다는 전체적인 설계나 단계적 기획/계획 중심으로 답변해줘.

# 답변

아주 체계적으로 고민하셨고, 실제 게임 개발에서 자주 마주치는 확장성/유지보수 문제까지 잘 짚으셨습니다.

아래에 각 항목별로 설계 방향과 실무적 제안을 드립니다.

---

## 1. 크래프팅 테이블과 플레이어 상호작용 (UI 팝업)

- **테이블 씬**은 Area2D(또는 StaticBody2D) + Sprite + CollisionShape2D로 구성
- 플레이어가 테이블의 Area2D에 진입하면 "상호작용 가능" UI(예: E키, 클릭 등) 표시
- 입력 감지(키/마우스) 시 CraftingUI를 인스턴스화하여 팝업(보통 CanvasLayer나 UI Root에 add_child)
- **확장성:** 테이블 종류별로 별도 씬을 만들거나, 공통 베이스(예: [CraftingTable.gd](http://craftingtable.gd/))에서 종류/텍스처만 바꿔 사용

---

## 2. 크래프팅 테이블 종류 관리 및 레시피 분류

### **가장 유지보수/확장에 좋은 방법**

- **Recipe 리소스에 "테이블 타입" 필드 추가**
    - 예: `@export var table_type: String = "anvil"`
    (예: "anvil", "armor_shelf", "furnace" 등)
- **테이블 종류가 늘어나면 Recipe 리소스의 table_type만 추가/수정**
- **CraftingUI는 테이블 종류를 받아, 해당 table_type에 맞는 레시피만 필터링해서 RecipeList에 표시**
- **RecipeIngredient, ItemData, EquipmentData 등은 레시피 분류에 직접 관여하지 않음**
(즉, 레시피의 "소속"만 Recipe에서 관리)

### **장점**

- 테이블 종류가 늘어나도 Recipe 리소스만 수정하면 됨(유지보수 용이)
- 레시피가 여러 테이블에서 사용될 경우, table_type을 배열로 확장 가능
- 레시피와 테이블의 관계가 명확하게 분리됨

---

## 3. Slot 씬의 재사용/스크립트 제거

- **Slot 씬을 아이템 표시용으로만 쓸 경우, 스크립트 없이 인스턴스화해도 무방**
    - 단, 아이콘/수량 표시 등 최소한의 기능만 필요하다면
    별도의 "DisplaySlot" 씬을 만들어도 됨
- **Slot.gd의 드래그/드랍 등 불필요한 기능이 많다면,
크래프팅 UI 전용의 경량 Slot 씬을 따로 만드는 것이 유지보수에 더 좋음**
- **임시로 스크립트만 제거해서 써도 되지만,
장기적으로는 크래프팅 전용 Slot 씬을 별도로 두는 것이 권장**

---

## 4. 제작 가능 여부(RecipeList에서 GreyedOut)

- **CraftingUI에서 레시피별로 제작 가능 여부를 판단**
    - 인벤토리의 재료 합산, 결과물 슬롯 여유 체크
- **ItemList의 set_item_disabled/set_item_custom_fg_color로 시각적 표시**
    - 제작 불가: 회색, 클릭 불가
    - 제작 가능: 흰색, 클릭 가능

---

## 5. CraftingUI 스크립트 구조

### **권장 구조**

- **CraftingUI(PanelContainer)에 메인 스크립트 부착**
    - 레시피 목록, 재료/결과물 표시, 제작 버튼 활성화 등 전체 로직을 담당
    - 하위 노드(RecipeList, GridContainer 등)는 신호(signal)나 메서드 호출로만 연결
- **하위 노드에 별도 스크립트 부착은
복잡한 UI(예: 드래그/드랍, 커스텀 동작 등)가 필요할 때만 고려**
- **대부분의 크래프팅 UI는 루트 스크립트 하나로 충분히 관리 가능**
(유지보수/확장에 유리)

---

## 6. 유지보수/확장성 평가

- **Recipe 리소스에 table_type 필드로 분류**
→ 테이블/레시피 추가/변경이 매우 쉽고, 데이터 중심 관리 가능
- **Slot 씬 분리**
→ 불필요한 기능 제거, UI별로 최적화 가능
- **CraftingUI 루트 스크립트 집중**
→ 전체 흐름 파악/유지보수 용이

---

## **정리**

- **Recipe 리소스에 table_type 필드 추가**
(테이블 종류별 레시피 분류/유지보수에 최적)
- **Slot 씬은 크래프팅 전용 경량 버전 별도 제작 권장**
- **CraftingUI 루트 스크립트에서 전체 로직 관리**
- **ItemList의 set_item_disabled 등으로 제작 가능 여부 표시**
- **테이블 종류/레시피 추가/변경이 매우 쉬운 구조**
